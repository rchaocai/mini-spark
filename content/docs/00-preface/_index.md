# 前言

## 当前大数据学习的难点？

目前，大数据领域的组件层出不穷，其中分布式计算是核心一部分，其中光分布式计算框架就很多，比如mapreduce，storm，spark和flink等，但无论框架如何变化，其底层逻辑始终相通，比如数据分片、任务编排、数据本地性等。因此一个高效的学习路径是选定一个成熟的框架深入钻研，比如 Spark。然而当你真正打开 Spark 官方文档时，扑面而来的往往是海量代码——光是让环境顺利运行就足以耗尽耐心。经过十余年的演进，Spark 已发展为一个极其复杂的系统，学习者若试图从头啃起，很容易在层层胶水代码中迷失方向，尚未触及核心便已深陷细节的泥潭。由此看来，更优的策略是跳过繁杂的现代表象，回溯框架最初的设计雏形，从最朴素的原型入手，理解其本质思想，再循序渐进地叠加现代特性。这样的学习方式，反而能事半功倍，直指要领。

多年前初识 Spark 时，我已能照教程跑通 WordCount，却始终说不清它究竟“快”在哪里。彼时，“内存计算”四个字挂在嘴边，可我分不清它的本质——MapReduce 也将数据加载到内存中执行算子，Spark 同样要经历 Shuffle，为何基准报告里 Spark 却总能遥遥领先？我也常提“惰性求值”，却讲不出那个“惰”字究竟何时才被打破，计算的扳机到底在哪个瞬间扣下。坦白说，若不深究原理，只照 API 编码，大多数任务也能完成。可那种“笼罩在迷雾中”的感觉始终挥之不去，像隔着毛玻璃写代码，心里发虚。

好在这份心虚并未消散，而是成了求知的原动力。经过多年研习，加上研究生阶段在分布式领域的系统输入，那些曾经模棱两可的概念终于渐次清晰。如今我终于有了底气去梳理、去输出。

## 为什么要"从零手写"？

这是spark系列的第一篇文章，写这些文章的原因，一方面是想分享，另一方面也是为了梳理自己的知识体系。近几年接触的知识太多太杂，尤其在 AI 辅助编程成为常态之后，我觉得对一名大数据开发者而言，真正重要的不再是死磕代码细节，而是领悟与理解——把道理想透，把代码交给 AI 去实现，这才是更高效的学习姿态。

市面上讲解 Spark 的书不算少，我翻过其中很多，不乏佳作。但它们大体可分两类：一类重实用，教你怎么用 Spark——API 怎么调、SQL 怎么写、参数怎么优化，这类书能帮你快速上手，属于“知其然”；另一类是源码剖析，数量较少也更重要，它试图回答“知其所以然”，但这类书籍一般会陷入代码中，最终只告诉你这段代码在干什么，却没有回答为什么这么做。此外其学习门槛比较高，初学者往往刚翻开就被源码的环境搭建劝退，即便硬着头皮读下去，也容易在庞杂的源码中迷失方向，难以坚持到底。

这些年我越来越相信一件事：**想真正搞懂一个系统，最好的办法，就是亲手做出一个来。**

当然，这不是要我们从头再造一个比 Spark 更强的引擎——既不现实，也没必要。我想说的是，当你用几千行代码写出一个能跑的 mini-spark，亲手把数据切成分片，亲手把计算逻辑发送到数据所在的地方，再亲眼看着一次 Shuffle 将中间结果落盘——那些曾经高悬云端的概念，会忽然变得朴素、具体、触手可及。

“原来 RDD 是这回事。”
“原来 Stage 只是一道边界。”
“原来 Spark 的容错不是魔法，而是血缘的必然。”

## 为什么是 Java？

这篇专栏的代码示例，我决定用 Java 来实现，一定会有人问：Spark 本身就是 Scala 写的，PySpark 又那么流行，为什么偏偏要用 Java 来"再造一遍"？

理由有三。

第一，用简单的编程语言。 这个专栏是写给"只会简单 Java"的人的，而在我看来Java相比于Scala容易理解很多，而且应用更广也利于大家的就业。如果换成 Scala，你在读懂第一个 RDD 之前，就得先被 implicit、for 推导式、上下文界定这些 Scala 独有的概念绊倒——那意味着在"理解 Spark"之外，你还得同步攻克一门新语言。这违背了我们从头到尾坚持的一条原则：一次只增加一个认知负担。 注意力是最稀缺的资源，既然你已经有了基本的 Java 基础，我们就把全部精力留给真正重要的东西——Spark 的设计思想，而不是语法查表。

**第二，要的是 Scala 的"形"，不是它的"难"。** Scala 之所以适合写 Spark，靠的其实是几样东西：用 `case class` 描述数据、用 `sealed trait` 表达"只有这几种"的封闭类型、把函数当参数传来传去。而 Java 17 恰好补齐了这几样——`record` 就是 `case class`，`sealed` 就是 `sealed trait`，再加上 Java 8 起就有的函数式接口。换句话说，我们用 Java，照样能写出几乎和 Scala 一样清爽、一样贴近 Spark 源码的代码，却不必背上 Scala 的复杂包袱。这也是为什么到了第 11 章，当我们打开真实的 Spark 源码时，会发现它和你写的"小玩具"几乎一一对应——因为我们用的，是和它同根同源的 JVM 语言。

**第三，Python 适合"用" Spark，不适合"造" Spark。** 用 Python 写 PySpark 很顺手，可它离 Spark 的内核隔了一层：它没有 JVM 那套静态类型，也很难干净地表达 RDD、Dependency、Task 这些内核抽象。拿 Python 讲"怎么用 Spark" 挺好；但讲"Spark 内部到底怎么运转"，Java 要直接得多。

一句话总结就是**用你性价比最高的语言（Java），去逼近 Spark 母语（Scala）的样子，从而把脑力和注意力加权到分布式计算的思想上，而不是编程语法上。**

## 写给谁？

- 你会写**基础的 Java**：变量、循环、集合、简单的类。这就够了。
- 你**不必**懂 Scala，不必懂数据库，不必懂分布式。这些我都不假设。
- 用到的进阶特性——函数式接口、`CompletableFuture`、`record`、`sealed`——都会在用到时，用一小节慢慢铺垫。

如果你已经是 Spark 老手，可以用来消遣，当作一到可口的零食；若你像曾经的我一样，"会用却想知道为什么"，那这个专栏就是为你写的。

我们用 12 章，一步步搭出一个属于自己的 mini-spark：

从一行最朴素的 WordCount 起步，你会先尝到"数据太大、单机扛不住"的真实焦虑；随后我们把数据分块、把代码送到数据那里；接着你会撞上第一个"顿悟时刻"——原来惰性求值并不是什么精巧的设计模式，它不过是几个嵌套的迭代器，自然连成了一条流水线。

最后，我们会打开真实的 Apache Spark 源码，把它和你刚写完的"小玩具"并排放在一起——你会看到，顶尖开源项目的核心骨架，和你在本专栏里一笔一画写出来的，几乎一模一样。

那一刻，Spark 就不再可怕了。

## 怎么读这本专栏

- 先看正文，知道这章要实现什么、为什么这么设计。
- 然后看代码，直接打开对应章节的目录，编译、运行、随手改改。

整个专栏基于 JDK 17：我们会用到 `record` 和 `sealed`。别被名字唬住，它们只是让 Java 写起来，更像它那位 Scala 表亲而已。

## 最后

写这篇专栏的初衷就是为了对过往知识的梳理，也是我重新学一遍 Spark 的过程。每把一个复杂的东西拆到足够简单，我都会生出一种"原来如此"的释然。希望这篇专栏，也能把同样的释然交到你手上。

如果读完之后，你面对 Spark 时，对各种专业名词少了一分畏惧，多了一分"我也做得出来"的底气——那我的分享就不算白费。

好了，就让我们从一行 WordCount 开始吧。
