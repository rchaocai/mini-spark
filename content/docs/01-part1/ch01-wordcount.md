# 第 1 章 · 从 WordCount 开始

> 💻 本章完整代码：[GitHub 查看](https://github.com/rchaocai/mini-spark/tree/main/ch01-wordcount)
>
> 构建运行：`mvn -pl ch01-wordcount compile exec:java -Dexec.mainClass=com.sparklearn.WordCount`

在前言的最后，我说"让我们从一行 WordCount 开始"。这不是一句客套——它是本专栏真正的起点。因为要理解 Spark 为什么长成今天这个样子，没有比"数词频"更合适的入口了。

数词频（WordCount），大概是大数据世界的"Hello World"。它问的问题简单到像开玩笑：给你一段文本，数一数每个词出现了几次。

别觉得这问题土。你每天打交道的很多东西，拆到底层，干的就是这件"数次数"的活：

- **搜索引擎**：你敲一个词，它得数一数这个词在多少篇网页里出现过、各出现过几次，才好决定谁排在前面；
- **输入法联想**：能猜到你下一个字想打什么，靠的正是"哪个词更常用"这张词频表；
- **热搜、词云**：今天哪个词被提到最多，数一数就知道了；
- **网站后台**：看一篇新闻被点了多少次、某件商品卖了多少件——数的是同一样东西。

它们的共同点，是 Spark 文档里常说的那类典型算式：按某个维度把一堆东西归拢到一起，数一数各出现了几次（或加一加、求个平均）。MapReduce 的那篇开山论文，举的第一个例子也正是数词频。

> [!INFO]
> **MapReduce 是什么？**
>
> 既然提了 MapReduce，就花一小段认识它——本书后面的 `map`、`reduceByKey` 这些算子，名字就是从它这里继承的。
>
> MapReduce 是 Google 在 2004 年提出的一套**分布式编程模型**（论文标题直译就是《MapReduce：简化大规模集群上的数据处理》）。它的厉害之处不在多高深，而在于：**你只管写两个函数，剩下的并行、容错、数据怎么搬到哪台机器，全交给框架。**
>
> 这两个函数，名字就写在这套模型里：
>
> - **Map（映射）**：读进一条数据，吐出若干 `(key, value)` 对。放到 WordCount 里，就是把每个词变成 `(spark, 1)`、`(hello, 1)`……一条进来，一个 `(词, 1)` 出去。
> - **Reduce（归约）**：按 `key` 把所有 `value` 收拢到一起，做一次汇总。WordCount 里，就是把同一个词所有的 `1` 加起来——`(spark, 1)(spark, 1)(spark, 1)` 变成 `(spark, 3)`。
>
> 把这两步套回 1.1 节那段代码，几乎一一对应：那两层 `for` 循环里"切词、吐 `(word, 1)`"就是 Map，`HashMap` 里"按词累加"就是 Reduce。换句话说，你早就会写 MapReduce 了，只是没用这个名字。
>
> 不过它有个短板：两个 job 之间想复用数据，只能把中间结果**写回磁盘**、下一轮再读出来——这正是 1.5 节那个"更狡猾的问题"。Spark 的内存缓存，就是冲着这个短板去的。


## 1.1 先来数数词：朴素版 WordCount

假设我们有这样两行文本：

```
hello spark hello world
spark is fast spark is fun
```

想数清每个词出现几次，用你会的 Java，会怎么写？大概是这样（对应代码 `ch01-wordcount/.../WordCount.java`）：

```java
List<String> lines = Arrays.asList(
        "hello spark hello world",
        "spark is fast spark is fun");

Map<String, Integer> counts = new HashMap<>();
for (String line : lines) {
    for (String word : line.split(" ")) {
        counts.put(word, counts.getOrDefault(word, 0) + 1);
    }
}

counts.forEach((word, n) -> System.out.println(word + "\t" + n));
```

短短十几行，干的事很直白：把每一行按空格切成词，再用一个 `HashMap` 记账——每碰到一个词，就把它的计数加一。`getOrDefault(word, 0)` 的意思是："这个词之前出现过吗？出现过就用旧计数，没出现过就当 0。"

运行一下，你会看到类似这样的输出（`HashMap` 没有固定顺序，你的输出顺序可能不同）：

```
spark	3
hello	2
is	    2
world	1
fast	1
fun	    1
```

看，你已经写出了一个能用的 WordCount。

先停一下，享受这个小成就。别急着往难的地方想——此刻这段代码工作得好好的，它没有任何问题。

它唯一的"问题"，要等到我们把输入变得很大、很大之后，才会暴露出来。

## 1.2 然后，把数据换成一份 50GB 的日志

现在，假设输入不再是两行，而是一份 **50 GB** 的访问日志——比如一个中等流量网站跑了小半天攒下来的那种。

50 GB，听起来不像天文数字，但对一台笔记本来说已经够呛了。我们来算一笔账。

先看**内存**。一台还算不错的笔记本，内存大概是 16 GB。而 1.1 节那段朴素 WordCount，第一步就是把所有行一股脑塞进一个 `List`。要把 50 GB 的数据塞进 16 GB 的内存，50 ÷ 16 ≈ **3 倍**——根本装不下。程序会直接抛出 `OutOfMemoryError`，连开始算的机会都没有。

再看**速度**。就算数据真能装下（比如你换了台 64 GB 的大机器——50 GB 勉强塞得下，但余量已经不多了），目前的程序是单线程，只会使用一颗 CPU 核在算。乐观估计，单核做"读、切词、更新计数"这种活，每秒能处理大约 100 MB。50 GB ÷ 100 MB/s ≈ 500 秒，差不多 **8 分钟**。而你的笔记本明明有 8 颗、16 颗核，它们却全程在旁边看戏。

于是，我们结结实实撞上了两堵墙：

- **墙一：存不下。** 数据大到单机内存装不下。
- **墙二：算得慢。** 一颗核算得太慢，机器上其余的核却用不上。

这时候你再回头看看那段 WordCount，会觉得它天真得可爱——它默认所有数据都能舒舒服服地躺进内存，还默认全世界只有一颗核在干活。可真实世界的数据从来不会这么体贴。

## 1.3 第一面墙怎么破：把数据分块

先解决"存不下"。

思路朴素得像吃一块太大的蛋糕：你不会一口吞下整块，你会**切成小块，一块一块吃**。

数据也一样。50 GB 装不进一台 16 GB 笔记本的内存，那就把它切成许多份，每一份都小到能舒舒服服放进内存。一份一份地读、一份一份地算，算完一份就丢掉，再去拿下一份。

这个"一份"，在 Spark 里有个专门的名字，叫**分区（Partition）**。

先别被这个英文词吓到。它就是"切出来的一块数据"，仅此而已。把一整坨大数据切成一个个分区，是处理大数据最古老、也最本能的一招——**分治**。

到这里，"存不下"这面墙，我们撬开了一条缝：不一次性吞，分块慢慢嚼。

## 1.4 第二面墙怎么破：让计算并行

再解决"算得慢"。

切成块之后，要是你仍然只有一颗 CPU 串行地一块块算，那"8 分钟"的账并没有变，而且其余的核照样闲着。怎么把这些闲置的算力用起来？

先看看手头这台机器：它明明有 8 颗、16 颗核。让这些核一起上、各算一个分区，确实能快不少。可一台机器的核再多也就那么几颗，内存也就那么大；而分区可能有成百上千个。50 GB 或许还能靠把单机的核榨干勉强应付，可数据不会停在 50 GB——再涨个十倍，单机的核数和内存就撞到天花板了。

既然一台不够，那就加机器：把整批分区交给一个集群来并行——100 个分区撒到 100 台机器上同时算，原来单核要 8 分钟的活，摊下来可能几十秒就完事。这才是真正撬动"算得慢"的那根杠杆——**并行**，而且是跨机器的并行。回想 1.1 节那段代码，它默认全世界只有一颗核在干活；只要让计算在分区上同时铺开，那些闲置的核就不会再看戏了。

可一旦跨了机器，一个绕不开的问题就冒出来了：**数据和计算，在这些机器上到底怎么摆？** 这事其实有两派做法，都成立，也都在用：

- **存算一体：数据在哪，计算就去哪。** 数据用一个分布式文件系统（最经典的就是 HDFS）切成块、存多份副本，散落在集群各台机器上；计算程序也跑在这些同样的机器上。要算某块数据时，框架尽量把任务派给"正好存着这块数据"的那台——把代码发给数据，数据不用动。这就是**数据本地性（data locality）**，也是经典 Hadoop / Spark 的默认姿势。
- **存算分离：计算节点主动去拉数据。** 数据统一躺在独立的存储层（比如对象存储 S3、数据湖），计算则是单独的、能随时扩缩的集群，需要哪块数据就过网络拉过来。这正是**当下云上的主流趋势**：存储用便宜的大池子、计算按需起停，两者互不绑定；代价是数据每轮都得过一遍网络。

> [!INFO]
> **HDFS 是什么？**
>
> HDFS（Hadoop Distributed File System，Hadoop 分布式文件系统）是 Hadoop 那套工具的"地基"——专门把大到单机放不下的文件，切成一个个**块（block）**，每块再存上**多份副本**（默认 3 份），分散存在集群的各个 **DataNode** 上。
>
> 它和"存算一体"天生一对：正因为同一份数据有副本、散落在各台 DataNode，计算框架才总能在"离数据最近"的那台机器上把活儿干掉——这正是数据本地性的底气。本书的 mini-spark 不会真去实现一个 HDFS，但会沿用"数据分片 + 多副本散存"这套假设。

两派做法中**并行这根杠杆是一样的**——差别只在"数据要不要经过网络"。存算一体省网络、讲本地性；存算分离换来弹性，代价是每轮多搬些数据。

我们的 mini-spark，走的就是**存算一体**这条经典路线：假设数据已经分片存在各个计算节点上，于是顺理成章地推出那句贯穿全书的话——**把代码发给数据，而不是把数据发给代码。** 这不是大数据唯一的解，却是理解 Spark 调度、本地性、容错这些设计最稳妥的起点。

> [!INFO]
> **这里说的"代码"，到底指什么？**
>
> 别误会成"把你写的整个程序原封不动发到存着数据的那台机器上去"。不是这个意思。
>
> 真正发过去的，是**你要在某一块数据上执行的那一小段计算逻辑**。拿 WordCount 举例，就是"把每一行按空格切成词、对每个词记一笔 `(词, 1)`"这一小段活。框架把这段逻辑连同它需要的东西打包成一个**任务（task）**，送到存着那个分区的机器上，就地执行。
>
> 所以"把代码发给数据"更贴切的理解是：**把"对这块数据要做什么"这件事，送到数据所在的机器上就地干完。** 至于统筹、调度、提交作业的那部分程序，一直留在原地，并不跟着搬。

所以，大数据计算最核心的一个直觉，也是后面所有"算子"真正的源头，可以浓缩成一句：**把计算拆到一个个分区上，让它并行地铺开。** 至于每个任务具体跑在哪台机器——是把代码发给数据（存算一体），还是把数据发给代码（存算分离）——那只是调度层的策略，并不影响算子本身。让算子成立的，是分区上的并行，而不是"非得挨着数据跑"。

## 1.5 墙后还藏着一个更狡猾的问题

现在，"存不下"和"算得慢"似乎都找到了出路——分块、让计算并行。

但别以为有了这两招，大数据计算就万事大吉了。这里还藏着一个更狡猾的问题。

分布式计算本身，早在 Spark 之前就有方案了。但它们有一个共同的短板：两个 job 之间想要复用数据，只能把中间结果**写回磁盘**，下个 job 再**重新读出来**。读、写、序列化、网络复制——每一轮都这么折腾一遍。偶尔一两次还能忍，可真实世界里的很多计算是迭代的：

- **逻辑回归**训练要跑几十轮，每轮都要过一遍全部数据；
- **PageRank** 在大图上反复迭代，一轮不够，要几十轮；
- **运维工程师**对着几 TB 的日志交互式地查——不是查一次，是一次又一次。

这三种场景的痛点是同一个：**它们都要反复读同一批数据。** 每轮都从磁盘重来，慢得不讲道理。如果能把数据留在内存，第二轮、第三轮直接拿——能快出一个数量级。

> [!INFO]
> **逻辑回归是什么？**
>
> 逻辑回归是最常用的**分类**算法之一——比如判断一封邮件是不是垃圾邮件、一个用户会不会点广告。它靠不断微调一组参数来"学会"判断。
>
> 它的训练用**梯度下降**，一轮一轮地调参；而**每一轮都要把整份训练数据从头到尾过一遍**，几十甚至上百轮才收敛。简化成伪代码是这样：
>
> ```text
> for epoch in 1..100:           # 要跑几十~上百轮
>     for 每条样本 in 训练数据:    # 每轮都重新过一遍"同一批"数据
>         用当前参数算预测值，再更新参数
> ```
>
> 重点看第二行：它**每一轮都在读同一批训练数据**。如果这批数据每轮都得从磁盘重新加载，就慢得让人抓狂——这正是"数据复用"要解决的。

> [!INFO]
> **PageRank 是什么？**
>
> PageRank 是 Google 创始人 Larry Page 提出的算法，给网页打"重要程度"分：一个网页被越多**重要**的网页链接到，它自己就越重要。
>
> 它要在整张网页链接图上**反复迭代**——每一轮，每个网页都根据"指向它的那些网页"的分数重新算一次自己的分数，几十轮才稳定下来：
>
> ```text
> repeat 几十轮:
>     for 每个网页 in 链接图:     # 每轮都重新扫一遍"同一张图"
>         新分数 = 所有指向它的网页的分数，按比例汇总
>     用新分数更新所有网页
> ```
>
> 同样，每一轮都在反复读**同一张图**。图越大、轮数越多，"把数据留在内存"带来的加速就越明显。

Apache Spark 正是为解决「数据复用」而生。我们从最简单的 WordCount 起步，但心里始终装着上面那三类"反复读同一批数据"的场景：**迭代式机器学习**（逻辑回归要跑几十轮）、**图计算**（PageRank 反复迭代）、**交互式查询**（运维一遍遍翻日志）。它们会在后面的章节里，一次次把咱们逼出更好的设计。

## 1.6 本章小结：两件法宝

回过头看，这一章我们其实没写什么高级的东西——没有 RDD，没有分布式框架，连一个新概念都没正式落到代码里。

但我们拿到了应对大数据的两件法宝：

1. **分块（Partition）** —— 分治：数据太大，就一块一块算。
2. **并行（算子的雏形）** —— 把计算拆到一个个分区上同时算（至于把代码发给数据还是把数据发给代码，只是调度策略，mini-spark 选了前者）。

这两条，朴素得几乎不需要解释，却几乎是 Spark 全部设计的出发点。

我故意在这一章没有引入任何抽象，是因为我相信：在真正理解一个复杂系统之前，先把它背后那个最简单的"为什么"想透，比记住任何名词都重要。

你现在已经把那个"为什么"握在手里了。

那么下一步，我们就来给数据安个家——把它变成代码里能拿来算的东西。下一章会请出一个叫 `ListRDD` 的小东西：它是这本书里你认识的第一个"数据"的代表，一个被切成若干分区、可以在上面并行算、还能反复复用的数据集合。
